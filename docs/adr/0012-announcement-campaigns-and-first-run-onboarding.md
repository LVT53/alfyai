# Announcement campaigns and first-run onboarding

AlfyAI will implement first-run onboarding as the first use of a reusable Announcement Campaign system, not as an onboarding-only modal. Campaigns are versioned, localized app-shell announcements with fixed slide layouts, curated desktop and mobile screenshots, per-user completion/dismissal state, and admin-managed draft/publish/archive lifecycle.

Published campaigns are immutable. Admins may create and preview draft campaigns in Settings, publish a new revision, and archive old published campaigns, but published content remains historically auditable and replayable. Corrections or follow-up announcements require a new campaign version or revision rather than editing the published record in place.

Published campaigns are stored as snapshots separate from editable draft state. The snapshot carries the frozen slide content and asset references used for user display and later audit, so changes to drafts, crop tools, or admin configuration do not alter what was published.

Only one campaign may auto-show in the app shell at a time. First-run onboarding takes precedence for users who have not completed or dismissed the current onboarding version. Release/update campaigns auto-show only after onboarding is no longer pending, and when multiple unseen release campaigns exist, only the latest published one auto-shows. The sidebar App Version Badge opens the latest published campaign in replay mode without resetting completion state or creating a popup queue.

The App Version Badge shows the Canonical App Version from package metadata rather than an admin-entered current-version setting. Release campaigns may link to that package version, but the deployed app version remains code/package metadata so the sidebar cannot drift away from the running build.

The badge displays a compact major/minor version such as `v1.4`, while the full package version remains available in hover text and admin metadata.

Campaign modals use a compact footer: Skip on the far left for non-final auto-shown slides, Back and Next on the right, and Finish replacing Next on the final slide. Skip closes the modal and marks the campaign version finished for that user; the final slide hides Skip. Primary progress actions are visually stronger than Skip/Back, and all clickable controls need clear hover states and pointer affordance.

The modal close button has the same state effect as Skip. This keeps dismissal lightweight because users can replay the latest published campaign from the App Version Badge.

Campaign identity is generated by the system from campaign type, version, and revision. Admins do not hand-edit campaign slugs or keys. Scheduling is out of scope for v1; campaign eligibility is based on published status, audience, campaign type, version ordering, and per-user state.

The app may seed a first-run onboarding draft template, but seeded templates are not auto-published and are not eligible for user display until an admin completes validation and publishes them.

First-run onboarding targets all users who have not completed or dismissed the current onboarding campaign version, including existing accounts. Ordinary audience settings do not narrow first-run onboarding v1.

Campaign authoring belongs in a dedicated Campaigns pane inside the admin Settings surface, separate from System and Users. The authoring UI should provide a fixed-layout slide editor with a live modal preview beside it, desktop/mobile preview modes, and English/Hungarian preview modes. Admins provide slide titles, short body copy, required screenshots, alt text, optional primary actions, and optional preference controls from a bounded set. Slide order should be managed directly through explicit move controls rather than drag and drop.

The admin Campaigns pane uses a production workbench layout. Desktop has a campaign list rail, a central editor, and a right-side exact modal preview. The list shows compact campaign rows with internal name, type chip, version, status chip, slide count, and updated or published date. The editor groups campaign fields, slide ordering, localized copy, image crops, optional actions, and setup-slide controls into clear bands without arbitrary nested cards. A sticky publish/checklist bar shows validation state and actions such as Preview in shell, Save draft, and Publish. Published and archived views are read-only snapshots with audit metadata, campaign analytics summaries, archive where applicable, and Duplicate as draft.

Campaign delivery uses a fixed responsive modal layout. Desktop uses a centered modal with the blurred overlay treatment established by the search modal, an 8px radius, subtle border, compact header, segmented progress, prominent 16:10 screenshot, localized copy, and fixed footer. Mobile uses a near full-screen modal or sheet with safe-area padding, the 9:16 mobile screenshot crop, scrollable content, stacked setup controls, and sticky footer.

V1 supports only setup and standard slide layouts. Both layout types still require desktop and mobile screenshots. Data disclosures use the standard layout rather than a special text-only variant.

Admins can preview draft campaigns in the real app-shell modal before publishing. Draft shell preview is admin-only, visibly marked as preview, and does not mutate per-user completion state or write campaign interaction analytics.

Campaign screenshots are curated real captures of released AlfyAI features, not live user-state screenshots and not invented mock panels. Each slide requires desktop and mobile screenshot crops using stable aspect ratios. The admin upload flow may accept larger source images and produce the required crops inside AlfyAI so the admin does not need to prepare exact-ratio assets externally.

The campaign crop ratios are 16:10 for desktop screenshots and 9:16 for mobile screenshots.

Each slide may use separate desktop and mobile source uploads before cropping, with optional source reuse when a single image works for both frames.

Cropping is handled in a dedicated admin crop modal rather than inline in the slide editor. The modal supports fixed-ratio crop boxes, zoom, repositioning, reset, and save actions for desktop and mobile crops.

Campaign assets are app-owned files outside the Knowledge Base. Admin routes handle upload and crop operations; published campaign snapshots reference asset ids. Draft assets are visible only to admins, while assets referenced by published campaigns may be served to authenticated campaign viewers.

Reusable campaign UI chrome is localized through the app i18n dictionary. Campaign-specific slide content is admin-authored in English and Hungarian and stored with draft and published campaign snapshots.

Publishing a campaign requires strict completeness validation. A draft cannot publish with missing localized content, missing desktop/mobile crops, invalid slide order, invalid preference controls, invalid action destinations, or missing type-specific requirements such as first-run setup and data disclosure slides.

Slide action destinations are restricted to an allowlist of internal AlfyAI routes in v1. Campaigns do not support arbitrary external URLs.

Campaign interaction analytics are minimal admin-facing events for delivery and engagement: auto-shown, slide viewed, completed, skipped or closed, replay opened, and setup preference changed. Events do not include free-form user content or heatmap-style tracking.

Campaign analytics are shown in the admin campaign detail/history surface as simple campaign-revision summaries, not in the general analytics surface in v1. (Note: the standalone Analytics tab was removed for normal users in ADR 0043; admin system analytics now lives under Administration. Campaign analytics remain in campaign detail/history regardless.)

First-run onboarding begins with an Onboarding Setup Slide for the most important user defaults: UI language, theme, model default, and AI style. The remaining slides are feature introductions and disclosures. The required data disclosure slide states that AlfyAI is not fully local and may process messages/files through configured providers and integrations while storing memory or analytics according to deployment configuration. It is an acknowledgement disclosure unless the product offers a real opt-out.

V1 preference controls are limited to the Onboarding Setup Slide. Other campaign slides may introduce features, disclose behavior, or navigate to allowlisted app surfaces, but they do not embed preference toggles.

The model default control includes a System Default Preference. Choosing System default means live inheritance from the administrator-configured default model, not copying the current default into the user's profile. Explicit user model choices remain personal overrides.

When introducing System Default Preference, existing users whose stored preferred model equals the configured default model migrate to inherited System default. Users with a different stored preferred model keep that model as an explicit override. New users start with System default.

Announcement Campaign v1 should be a production-ready release, not a demo prototype. It must include the user shell delivery path, admin authoring path, image cropping, slide ordering, publishing and archival flows, immutable snapshots, sidebar replay, durable per-user state, System Default model inheritance, EN/HU localization, and regression coverage for eligibility, publishing immutability, preference writes, dismissal, and responsive modal layout.

**Considered Options**

- Build a one-off onboarding carousel with local dismissal state.
- Store onboarding slides in code and update them only through deploys.
- Let admins create arbitrary rich modal layouts.
- Treat every deployed app version as an automatic popup trigger.
- Queue every unseen update campaign for users.
- Make published campaigns editable in place.
- Use a reusable campaign system with immutable published revisions, admin authoring, fixed layouts, and per-user state.
- Store published campaigns as snapshots separate from editable draft state.

We chose reusable immutable campaigns because onboarding, release announcements, and important disclosures share the same shell behavior, audit requirements, localization burden, and replay needs. A one-off modal would duplicate future update-popup work and make later release communication harder. Arbitrary rich layouts would make previews, accessibility, screenshots, and mobile behavior harder to guarantee. Automatic version-math popups and unseen campaign queues would create modal fatigue. Mutable published campaigns would break auditability and user trust because a dismissed campaign could later mean different content.

**Acceptance Scenarios**

- A user who has not completed or dismissed the current first-run onboarding sees onboarding after authenticated app load.
- Published first-run onboarding is eligible for existing and new users who have not completed or dismissed that onboarding version.
- A brand-new user does not see a release/update campaign immediately after onboarding in the same app-load flow.
- A user who has completed onboarding and has unseen release campaigns sees only the latest unseen published release/update campaign.
- A user never receives a queue of older release popups through auto-show.
- Clicking the sidebar App Version Badge opens the latest published campaign in replay mode even if the user already dismissed it.
- The App Version Badge displays the package-metadata app version rather than an admin-typed current version.
- The sidebar badge uses a compact major/minor display with full version available on hover or in metadata.
- Replay mode does not reset completion or dismissal state.
- Skip closes an auto-shown campaign and prevents that campaign version from auto-showing again.
- Closing the campaign modal with the close button behaves like Skip.
- The final slide uses Finish instead of Next and does not show Skip.
- Admins can create a draft campaign, edit slides, preview desktop/mobile and English/Hungarian variants, publish it, archive it, and view historical published content.
- Campaign authoring lives in a dedicated admin Campaigns pane rather than the general System pane.
- Admins can open a non-mutating app-shell preview for draft campaigns before publishing.
- Published campaign content cannot be edited in place.
- Published campaign content is read from a frozen snapshot rather than reconstructed from mutable draft fields.
- Publishing a correction creates a new campaign version or revision.
- Campaign keys are system-generated and not admin-editable.
- V1 does not require scheduling fields.
- A seeded first-run onboarding template may exist as a draft, but users do not see it until an admin publishes a valid campaign.
- Every slide requires English and Hungarian title/body copy.
- Every slide requires desktop and mobile screenshots with stable crops.
- V1 supports only setup and standard slide layouts; data disclosure slides use the standard layout.
- Desktop campaign screenshots use a 16:10 crop, and mobile campaign screenshots use a 9:16 crop.
- Admins can provide separate desktop and mobile source images for each slide before cropping.
- Campaign image crops are created in a dedicated crop modal with fixed-ratio crop, zoom, reposition, reset, and save controls.
- Screenshot assets are curated captures of released AlfyAI features and do not show private user data.
- Campaign assets are not Knowledge Base documents, chat attachments, or public static files.
- The crop workflow can accept source images larger or differently shaped than the final campaign frames.
- Slide order can be changed directly through explicit accessible move controls.
- First-run onboarding includes a setup slide for language, theme, model default, and AI style.
- V1 campaign preference controls only appear on the first-run onboarding setup slide.
- Choosing System default for model selection follows the current admin default model until the user chooses a specific override.
- Existing users whose stored preferred model matches the configured default migrate to System default; other existing model preferences remain explicit overrides.
- First-run onboarding includes a required data disclosure slide.
- Campaign publishing fails with field-level validation when required localized content, required crops, valid action destinations, valid preference controls, or type-specific requirements are missing.
- V1 campaign action buttons can only navigate to allowlisted internal app routes.
- Campaign interaction analytics record minimal delivery and engagement events without free-form user content.
- Campaign analytics summaries are visible from campaign detail/history rather than the general analytics surface.
- Onboarding feature slides teach released capabilities without creating projects, uploading files, changing admin settings, or mutating workspace content.
- User-facing campaign text, controls, actions, empty states, and errors are localized in English and Hungarian.
- Campaign-specific slide copy is stored as campaign content, while reusable controls and validation messages stay in the app i18n dictionary.
- Announcement Campaign v1 is fully functional end to end, with admin authoring, delivery, replay, auditability, localization, tests, and desktop/mobile layout verification.
- Campaign delivery uses a polished desktop modal and mobile sheet/modal layout with blurred backdrop, focus management, and non-overlapping responsive content.

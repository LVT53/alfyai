# Account Erasure Keeps Only Anonymous Aggregates

> **See also [ADR-0049](0049-analytics-excluded-users.md):** admins can additionally *exclude* still-identified users (including deleted-but-not-erased ones) from System Analytics dashboards without deleting rows — a presentation filter, distinct from the data removal described here.

Account Erasure removes person-linked local and app-controlled external account data, including analytics rows that retain user identity, email, name, conversation titles, or message identity. We keep only anonymous aggregate usage and cost totals after erasure, rather than pseudonymous per-user history, because retained person-shaped analytics would be easy to reidentify and would make deletion misleading despite preserving useful operational totals.

## Update (arch-hardening C7): analytics identity is no longer denormalized

The analytics rollups (`usage_events`, `analytics_conversations`) previously copied the user's **email and name** onto every row (`recordMessageAnalytics` / `recordConversationAnalytics`). That denormalization is what forced erasure to chase those tables for identity, and a frozen email/name snapshot could reidentify a user even after their `users` row was gone. Those two columns are now **dropped** (migration `1777140000083_drop_analytics_identity_denormalization`); the dashboard read model resolves display identity from the `users` table at **read time**. This strengthens the guarantee: the rollups are anonymous-by-construction, and a user whose `users` row is deleted surfaces only by an opaque `userId` (no email/name). Full erasure still deletes those rows outright — they retain `conversation_title` / `message_id` — but the anonymity no longer depends on that deletion happening.

Erasure ownership is also consolidated: the ordered quiesce → cleanup sequence and the **single user-scoped-table registry** now live in one module, `src/lib/server/services/account-lifecycle`. The registry is the one authoritative enumeration of every user-scoped table, classifying each as `cascade` (removed by the `ON DELETE CASCADE` from `users.id` — the real mechanism), `explicit` (the two analytics rollups, which have no FK to `users` and are deleted directly), or `detach` (shared content, per ADR-0031). It replaced the two hand-maintained delete-lists that used to shadow each other and the cascade. A per-table test seeds a row in **every** user-scoped table and asserts zero survivors after erasure.

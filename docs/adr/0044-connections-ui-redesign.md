# Connections UI Redesign

Five decisions for the Connections feature's user-facing surface, made together after a design interview because they interact: the composer control model, the settings layout, the per-connection detail surface, the email connect flow, and the privacy-policy surface. Supersedes the Phase-7 functional prototype (see [[connections-ui-redesign-pending]] context). The underlying backend (connect/read/write/locality/isolation) is unchanged and verified; this ADR governs presentation and the one interaction-model change it implies.

## Decisions

**1. Composer carries a single trust-the-assistant Connections Toggle, not per-capability controls.** The composer gets one **Connections** on/off icon in the bottom row beside the incognito toggle (not a plus-menu row, not per-capability rows). On (default) = the assistant may use the user's default-on Connections and *decides itself* when to reach for calendar/email/files/etc., exactly as it already decides when to call `research_web`; off = no Connections that turn. All per-capability granularity moves to Settings' default-on. Multi-account disambiguation is resolved by the assistant in chat + learned into memory, never in the composer. Rejected: per-capability composer toggles (the shipped prototype — cluttered the plus menu, made the user micromanage tool availability, and read as "over-thought"); a single plus-menu row (still adds a row; an icon in the existing composer action row is leaner and matches the incognito toggle already there); removing the control entirely (loses the user's per-turn "don't touch my accounts for this sensitive question" escape hatch, which the single toggle preserves cheaply). NOTE: this removes the per-turn/per-capability selection built in Phase-7 issue 7.2; the fail-closed server intersect (`resolveActiveCapabilities`) stays, now driven by the master toggle + default-on set rather than a client-sent per-capability list.

_Amendment (2026-07-14):_ the toggle's choice is now **remembered per conversation** (persisted in `localStorage`, keyed by conversation id) instead of resetting to on whenever the bound conversation id changes. The original reset-per-conversation implementation flipped the toggle back on across the draft→real conversation creation (e.g. when a model switch created the draft) and the post-send `/`→`/chat/[id]` navigation remount, so an off choice silently reverted — contradicting the user's explicit choice. The escape-hatch intent is preserved: brand-new drafts still default on; the off choice now persists for the conversation it was made in (and survives reloads). Implemented in `MessageInput.svelte`.

**2. Connections settings is a compact list, not fat cards.** One row per Connection: brand icon · account · capability mini-icons · status · quick icon actions (reconnect/disconnect). Rarely-touched controls (allow-writes, default-on, folder allowlist) do not appear up front. Rejected: the always-expanded prototype card (4-5× denser than every other Settings surface like the Memory card, which is one switch + a line of help — the source of the "messy" feeling); a two-column list/detail split (heavier build, mail-client feel the product owner did not want).

**3. Per-connection settings live in a centered, content-sized Connection Detail modal overlay.** Opening a row raises a modal on top of the list (it overlays, it does not replace the row or expand it inline). The modal is **centered and sized to its content** — a modest-width settings form with comfortable padding and internal scroll if it ever exceeds the viewport — NOT a full-height/full-screen sheet (an early implementation used a full-height sheet; superseded — the intent was always an overlay popup, per the product owner). The modal holds capabilities, default-on, allow-writes (its reversible/allowlist warning behind the shared `InfoTooltip`, not always-visible amber prose and not a new look-alike chip), the folder allowlist, and a quiet danger disconnect icon in the header row. Rejected: inline accordion expansion (keeps the list dense and jumpy); replacing the list with an in-place detail panel (loses the glance-able list); full-height sheet (too tall for a short settings form — dead space).

**4. Email connect is one tile that opens a multi-step wizard offering Alfy Email and Gmail & Other (IMAP).** The single Email entry opens a guided modal with two paths: **Alfy Email** — near-zero-config for on-box (Virtualmin) accounts, host/port/TLS predictable, user enters only email + password; and **Gmail & Other (IMAP)** — a Gmail-branded path (enable IMAP + app-password help) plus a manual IMAP fallback. Rejected: the raw manual-IMAP form as the only path (the shipped prototype — felt like configuring a mail client, the product owner's exact complaint, and offered no Gmail option); Gmail via Google OAuth (Gmail API is a Google "restricted" scope requiring the paid third-party security assessment — not worth it; IMAP + app-password is the pragmatic Gmail path).

**5. The privacy policy lives at a public `/privacy` route; the profile entry links straight to it (no in-app modal).** The policy is a public, unauthenticated `/privacy` page (so it can be submitted to Google's OAuth verification), and a compact row in the user profile simply navigates to that page. Entity: AlfyAI; contact: levente@alfydesign.com. Rejected: an in-app full-screen modal duplicating the page (an early build added one; the product owner asked to remove it — the standalone page is the single surface, and the profile row just points to it); modal-only (Google needs a public URL); duplicating the copy in two places (drift risk).

## Context

The Phase-7 Connections UI was built functional-first as a working prototype and verified for correctness, on the explicit understanding that a full visual redesign would precede deploy. A design interview surfaced that the prototype was usable but over-built: the composer's per-capability toggles were the main "very ugly" offender, and the settings cards were far denser than the rest of the app. The product owner also flagged real functional gaps behind the visual ones — no Gmail option and no easy path for the many on-box (Virtualmin) mailboxes, which the original plan intended as tiered email connect but only shipped as raw IMAP. Brand icons are needed because Lucide no longer ships brand glyphs (vendor from Simple Icons; use Google's official "Sign in with Google" styling for that button specifically). The privacy policy is a prerequisite for moving the Google OAuth app from Testing to Production (verification checks the URL). Cross-cutting cleanups fold in here: reuse the shared `InfoTooltip` everywhere (audit and unify any divergent look-alike chip, including on the Memory page), fix the name↔account spacing gap, and replace text buttons/labels with accessible icon buttons.
